# 我给妈妈写了一个炒股工具，顺便聊几个工程上的真实坑

> 背景：程序员，妈妈炒股。市面上的 App 不是信息太多就是操作太复杂，于是自己动手写了一个。从第一行代码到现在跑通完整链路，大概两个月。这篇文章不是教程，是这个过程中踩坑、做决策、反复推翻自己的记录。

## 一、为什么不用现成的

我妈炒股多年，水平属于"看新闻凭直觉"。她的痛点很具体：

- **券商 App 太复杂**。她要看的就是"我的持仓现在赚了还是亏了"、"今天有没有什么异动"，但 App 给她推一堆她看不懂的东西。
- **不会用电脑端工具**。同花顺、东方财富 PC 端对她来说学习成本太高。
- **没有人帮她判断**。她需要的不只是行情数据，而是"今天该不该注意什么"这个判断。

我是个程序员，能写代码。所以答案很自然——自己造。

但说实话，一开始我也不知道最终做成什么样。很多设计是在做的过程中被打脸打出来的。

## 二、最终做成了什么

先看结果。两个月后，这个工具能做这几件事：

**给妈妈用的（手机 H5 页面）**：
- 自选股实时行情 + K 线（日/周/月 + 分时）
- 资金流图表（主力今天买了还是卖了，近 7/30/90 天趋势）
- 盘面扫描（大盘指数 + 涨跌幅榜 + 板块榜，30 秒轮询）
- 持仓管理（语音录入："茅台买入价1680 100股"自动解析）
- 信号告警（7 条规则，触发后微信推送）

**给自己用的（开发者视角）**：
- 3 条产业链的成分股库（人形机器人 25 只 / 半导体 106 只 / 材料 41 只）
- 资金流 ratio 监控（不是绝对金额，是相对市值的比率）
- 收盘日报自动生成 + 微信推送
- 4 个定时任务（盘前预热 / 盘中监控 / 收盘日报 / 周报）

## 三、几个真实的工程坑

### 坑 1：第三方接口会在你最想不到的时候挂

凌晨 2 点。我在测监控的持续轮询，5 只自选股。突然东财的接口开始返回 Empty reply，连续 5 次全挂。

这个教训直接改变了架构设计。我在那之后加了一层 **FallbackAdapter**——主数据源（东方财富）挂了，自动切到备用源（腾讯财经）。不是什么高深的架构，但实战中确实救了场：

```
FallbackAdapter([EfinanceAdapter, TencentAdapter])
# 任意方法调用：先试东财 → 抛异常 → 试腾讯 → 都挂才返回 None
```

对应到代码层面就十几行 `contextlib.suppress`，但这个坑不踩一脚，你根本想不到要加。

**原则落地**：拉新失败不删缓存。数据库里有旧数据就给妈妈看旧数据，标注好"5 分钟前的数据"，总比给一个空白页面强。妈妈不需要知道后端发生了什么，她只需要看到数字。

### 坑 2："主力净流入 5000 万"这个阈值毫无意义

最早我的信号规则写的是绝对金额：主力净流入超过 8000 万就告警。用了一天就被自己否了。

问题在于：**同样 5000 万净流入，对茅台（流通市值 1.5 万亿）是 0.03bp 的噪声，对一个 100 亿市值的小票是 35bp 的异动——差了 1000 倍。**

用绝对金额做阈值，大盘股永远不触发，小盘股天天报警。

后来改成了 **ratio-based 信号**：主力净额 / 流通市值 = bp（万分之几）。5bp 是 WARN，10bp 是 CRIT。这一个改动让信号的误报率降了一个数量级。

这是那种"用过才知道不对"的设计。纸面上看绝对金额阈值简单直接，实战中完全不可用。

### 坑 3：缓存不是 TTL，是节流

很多人对缓存的理解是 TTL：数据过了过期时间就删掉，重新拉。

这个模型在这场景下有个致命问题：**如果拉新失败了怎么办？** TTL 过期了，数据被删了，接口又挂了，用户看到的是空白。

我最终用的是**节流 + 失败保留**的模型：

- **节流**：距离上次拉新不到 N 分钟就不拉新（控制频率，防封 IP）
- **数据可用性**：跟拉新是否成功完全解耦

数据库永远有数据，区别只是"5 分钟前拉的"还是"5 秒前拉的"。妈妈能看到时间戳，自己判断新鲜度。

```
CachedMarketDataAdapter
├─ 查 DB → 有数据？
│  ├─ 有 + 未到节流间隔 → 直接返回
│  ├─ 有 + 到了间隔 → 拉新 → 成功就更新 / 失败就保留旧数据
│  └─ 没有 → 拉新 → 失败就返回 None（冷启动没法兜）
```

每个缓存条目带**双时间戳**：`fetched_at`（我们什么时候拉的）和 `quote_ts`（数据自身的时间）。这两者分离后，"数据多新鲜"和"我们多久没刷新了"变成了两个独立问题。

## 四、架构长什么样

说出来你可能不信，整个系统就用了这些技术：

- **Python** + SQLAlchemy 2.0（ORM）
- **SQLite**（8 张表，单文件，好备份）
- **FastAPI** + WebSocket（后端 + 实时推送）
- **Vue 3 + Vite**（前端，手机 H5）
- **Server酱**（微信推送，免费 5 条/天）
- **cron 定时任务**（盘前/盘中/收盘/周报）

没有 Redis，没有 Celery，没有 Kafka，没有 Docker。单用户场景下这些全是过度工程。

核心的设计模式是**装饰器链**：

```
业务层（Monitor / Signals / CLI）
    ↓
CachedMarketDataAdapter   ← 加 DB 缓存 + 节流
    ↓
FallbackAdapter           ← 多源自动切换
    ↓
EfinanceAdapter           ← 东方财富（主力）
TencentAdapter            ← 腾讯财经（备用）
```

每层职责单一，每层独立可测。新增能力（比如以后加限流、熔断）就再套一层装饰器，不动业务代码。

数据类型层用 `@dataclass(frozen=True)`，金额一律 `Decimal`（不用 float），时间一律 `datetime + UTC`。这些是那种"不出事觉得多余，出了事才知道值"的约束。

## 五、实际体验

我妈妈现在每天的使用流程：

1. **9:30 开盘**：手机打开网页，看自选股行情
2. **盘中**：有异动会收到微信推送（比如"主力净流入超 10bp"）
3. **15:30 收盘**：自动收到收盘日报（板块资金流 + TOP 流入/流出 + 矛盾股）
4. **晚上**：看看 K 线和资金流历史趋势

她不需要打开券商 App 了。不是因为这个工具有多强大，而是因为它**只给她看该看的东西**，不推她不需要的信息。

## 六、一些坦诚的反思

做完这些功能后我意识到一个问题：**这个系统本质是一个数据管道**。

它能拉数据、缓存数据、展示数据、按固定规则触发告警。但它不理解"创新药板块为什么涨"，不会分析"这个股值不值得追"，更不能回答"妈妈该不该卖"。

这些都是需要**判断力**的事情，而判断力来自对政策、行业叙事、资金面博弈的综合理解——这是 LLM 的强项，不是几个 if-else 规则能替代的。

举个真实的例子。有次我用这个工具拉了创新药板块的数据，然后跟 AI 对话：

```
我：创新药板块今天怎么样？
系统数据：板块 +5.1%，180 只成分股全红，主力净流入 19.47 亿
AI：6 月 29 日医保目录初审通过 557 个药品，首次设立商保创新药
    目录，集采明确豁免专利期内创新药。这是政策底 + 估值底 +
    基本面拐点三重共振。具体推荐关注 PE<15、5 日涨幅<10%
    的补涨标的……
```

单独的系统跑不出这个结论（它只会报"主力净流入 19 亿"这个数字）。单独的 AI 也跑不出（它没有实时行情接口）。**两者拼在一起才有价值。**

所以如果让我重新来过，我会更早地考虑：**如何让 LLM 成为这个系统的大脑，而不只是把数据管道做得更完善。** 7 条写死的告警规则，不如一句自然语言灵活。K 线图表做得再好看，也不如一段"今天发生了什么、该关注什么"的文字总结对妈妈有用。

这不意味着数据管道本身没有价值——恰恰相反，它是 LLM 发挥作用的前提。没有可靠的实时数据接口，再强的模型也是空中楼阁。

**但正确的定位应该是：数据管道是基础设施，AI 才是面向用户的那一层。**

## 七、如果你也想做类似的东西

几个建议，都是用时间换来的：

1. **先跑通数据层再做 UI**。数据不稳定，UI 做得再好看也是空中楼阁。先确保能稳定拉到行情数据，有缓存、有降级，再考虑怎么展示。

2. **SQLite 起步，不要上 Postgres**。单用户场景下 SQLite 的性能绰绰有余，一个 `.db` 文件好备份、好迁移。等真的需要多用户了再升级不迟。

3. **用 Protocol（duck typing）不用 ABC**。Python 的 `Protocol` 配合 `@runtime_checkable`，做接口约束时比抽象基类灵活得多。切换数据源不需要改继承链。

4. **ruff + mypy --strict 从第一天就开**。类型问题在编译期发现，比线上跑出 bug 再查省太多时间。`Decimal` vs `float` 这种问题，mypy 能帮你拦住。

5. **测试覆盖核心逻辑**。我有 154 个测试（145 离线 + 9 网络），核心的数据拉取、缓存命中/降级、信号规则评估都有覆盖。前端的测试做得少，这是下一步要补的。

6. **不要一上来就追求完美**。我的第一版只有"拉一只股票报价"这一个功能。每个 milestone 都是在上一个能跑的基础上加的。先跑通再优化，先能用再好看。

## 八、写在最后

这个项目最终交付了什么？

从技术层面：**~14,000 行 Python + 154 个测试 + Vue 3 前端 + 4 个定时任务 + 微信推送**。

从用户层面：**妈妈打开手机就能看到自己持仓的行情，不用打开复杂的券商 App。**

但从更诚实地说：它还差一层"大脑"。数据管道再完善，如果没有能力把数据变成判断、把数字变成建议，它就永远只是一个"更聚焦的看盘工具"。

下一步我会尝试把 LLM 接进来——不是做一个 ChatBot 放在角落里，而是让 AI 成为系统的核心逻辑层：收盘后自动生成分析报告（不是"主力净流入 1.2 亿"这种机械描述，而是"今天发生了什么、板块在交易什么逻辑、妈妈该关注什么"这种真正有用的内容），主动推送到微信。

**给妈妈的工具，最终应该像一个懂行的朋友，而不是一个只会报数字的仪表盘。**
