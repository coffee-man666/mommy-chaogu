## 一、产品核心定位

### 1.1 利用本地可编程主机能力

团长明确指出，mommy-chaogu 的核心优势在于"**本地可定制**"和"**自主编程的主机能力**"。Mac mini 7×24 在线，本地直连行情 API（东方财富 / 腾讯），无中间环节，可以做到：

- **监控能力** —— 不依赖券商 APP 的标准化功能，可以自己编程定义监控逻辑
- **推送能力** —— 不经过券商复杂信息流，自主推送（Server酱）
- **数据直连** —— 直接调上游 API，延迟远低于券商转发

### 1.2 解决券商 APP 的三大痛点

| 痛点 | 描述 | 我们的解法 |
|---|---|---|
| 信息太多 | 每屏几十个模块，新手找不到重点 | 单一屏幕只展示关心的数据 |
| 操作太复杂 | 看一个东西得点好几下 | 一步到位，关键信息压在一屏 |
| 延迟高 | 推送 + CDN 中转 | 本地直连 + 后台 5 秒轮询 |

**核心价值主张**：用**简单的方式**获得**高效的数据**。

## 二、两大核心使用场景（重要！）

团长在对话中明确提出了**两类用户场景**的区分：

### 场景 A：观察盘面（发现新机会）

**目标**：扫盘、找热点、看趋势、捕获新标的

**信息需求**：
- 今天哪个最热？
- 资金在往哪里走？
- 整体有什么变化？
- 涨跌幅排行
- 资金流排行
- 行业板块排行

**信息特点**：
- 粗粒度（看到的是"哪只"，不是"怎么走"）
- 扫描型（榜单、对比）
- 不需要太密集更新（5-10 分钟看一次即可）

### 场景 B：管理仓位（细节决策）

**目标**：盯涨跌、看盈亏、决定买卖

**信息需求**：
- 持仓涨跌情况
- 盈亏金额
- 具体某只仓位的细节（成本价、股数、技术面）
- 是否需要技术面调整
- 主力资金动向

**信息特点**：
- 细粒度（看到的是"怎么走"）
- 集中（单只股的多个维度）
- 实时（5 秒轮询）

### 设计原则：让两个场景无缝切换

**不能把"扫盘"和"盯盘"混在一个界面里** —— 信息密度、交互方式都不同。需要清晰的导航把它们分开。

## 三、改进界面的具体方向

### 3.1 持仓管理是第一步

- 团长明确说："**专注持仓管理也很重要**"
- 目的："**通过自动化来帮助我妈更好的监控全局**"
- 目标：让妈妈"**可以放心看新的交易机会**"

### 3.2 录入要简单

团长关注过录入体验，提出了几个改进方向：

1. **语音录入**（已实现）—— 浏览器原生 SpeechRecognition API（webkitSpeechRecognition），iOS Safari / 微信内置浏览器都支持
2. **OCR 拍照批量录入**（未来考虑）—— 拍券商持仓截图，OCR 提取代码+股数+成本价，批量录入
3. **微信小程序**（未来考虑）—— 终极形态，原生支持语音/拍照/扫码

> 团长对话决策：**先走语音**，OCR 和小程序作为后续

### 3.3 资金流要看图

团长提出："资金流可不可以用图？"
- 日内分时折线图（SVG）
- 历史柱状图（SVG，零线居中，红正绿负）
- 累计汇总卡片（5 个维度：主力/超大单/大单/中单/小单）

### 3.4 隐藏不必要的信息

团长要求："**去掉盘口信息**"
- 5档盘口对扫盘和盯盘都不是最关键的
- 占用屏幕空间，移动端体验差

### 3.5 顶部报价区信息量太少

团长反馈："**前信息量太少**"
- 现有：今开/昨收/最高/最低/成交量/成交额/换手/量比/PE/主力
- 可以加：52周高/低、市值、EPS、股息率、所属行业

**盘面 Tab** 内容（取代现有"行情"）：

### 4.2 详情页变"决策驾驶舱"

详情页不再从上到下堆，而是顶部核心指标常驻 + 多 Tab 内容：

```
┌─────────────────────────┐
│ 1168.63  -1.30%   ⏰     │  ← 顶部永远固定
│ ┌──┬──┬──┬──┐          │
│ │开│收│高│低│          │
│ └──┴──┴──┴──┘          │
├─────────────────────────┤
│ K线 │资金│成交│新闻     │  ← Tab 切换
├─────────────────────────┤
│ [当前 Tab 内容]          │
└─────────────────────────┘
```

持仓时详情页多一个盈亏浮窗常驻顶部。

### 4.3 已确定的优先级

**短期先做方案 A 的"盘面 Tab"**：
- 涨幅榜 / 跌幅榜（数据易得）

- 主力净流入榜（需新增缓存字段）

**再做方案 B 的详情页驾驶舱**：
- 顶部核心指标常驻
- 内容 Tab 切换（K线/资金流/成交/新闻）

### 4.4 待讨论

## 五、技术决策记录

### 5.1 K线部分

- **图表库**：klinecharts 9.8（金融专用，gzip 后 ~53KB）
- **数据源**：东方财富 push2his.eastmoney.com
- **缓存策略**：SQLite 永久缓存，按 `(code, interval, adj_type, trade_date)` 主键去重
- **关键修复**：指标创建要判重（避免切换周期叠加多张成交量图）
- **详细文档**：`docs/KLINE-SPEC.md`

### 5.2 资金流部分

- **数据源**：efinance `get_today_bill` / `get_history_bill`
- **缓存**：SQLite，按 `trade_date` 存储
- **重要修正**：efinance 返回的是**到该时刻的累计值**（不是增量），累计应取最后一条
- **前端图表**：纯 SVG 实现（零依赖，移动端友好）
- **配色**：A 股红涨绿跌（流入红/流出绿）

### 5.3 持仓部分

- **数据模型**：Position + PositionAdjustment（支持加减仓/分红）
- **成本计算**：加权平均（含 buy/sell/dividend 调整）
- **录入**：表格 + 语音输入
- **首页联动**：持仓快览条 → 点击跳持仓页

### 5.4 架构原则

- **接口先行**：所有数据源走 `MarketDataAdapter` Protocol
- **降级优先**：第三方接口挂时 graceful fallback（主源 → 备源）
- **缓存优先**：DB 有就用，没有才拉新
- **dataclass + Decimal**：金额一律不用 float

## 六、K线修复 bug 记录

- **bug**：切换 K 线周期时，每点一次就多一个成交量副图
- **原因**：`chart.createIndicator('VOL')` 每次切换都调用，klinecharts 不判重
- **修复**：用 `isFirstInit` 标记，指标只在首次初始化时创建

## 七、产品哲学

1. **简单胜于复杂** —— 一个屏幕能看清的不分两个
2. **本地优于云端** —— 直连 API 快于券商转发
3. **自动化优于手动** —— 规则推送胜过人工盯盘
4. **聚焦优于堆砌** —— 只展示关心的，不展示能展示的
5. **场景优于功能** —— 按使用场景组织界面，而不是按功能模块
